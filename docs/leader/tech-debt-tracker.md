# EvoGenesis 技术债跟踪

## 当前状态

日期：2026-05-16

本文件用于记录 Leader 视角下的架构风险、延期决策和后续清理项。当前项目已完成 `v0.1 RNA 世界基础版`，下一步应先进行 `v0.1.1 运行体验修正`，再进入 `v0.2 前生命化学`。

## 技术债清单

| ID | 项目 | 状态 | 风险 | 处理建议 |
| --- | --- | --- | --- | --- |
| TD-001 | `PLAN.md` 曾混入 v0.1 详细设计 | 已处理 | 概要设计与详细设计边界不清，会导致 Builder 不知道以哪个文档为准 | 已将概要保留在 `PLAN.md`，v0.1 细节迁移到 `docs/builder/designs/V0.1-design.md` |
| TD-002 | 事件系统需要从首版建立结构 | 基础已实现，需增强表达 | 如果事件只写泛化文本，后续里程碑、趋势、灾难事件仍会难以解释 | `v0.1.1` 增加中文名、具体对象、事件证据快照 |
| TD-003 | 随机源和快照恢复必须统一 | 基础已实现，需保留回归测试 | 快照恢复后若随机状态不连续，会破坏复现实验价值 | 后续修改运行控制时必须继续验证随机状态连续 |
| TD-004 | 指标计算与模拟规则可能耦合 | 基础已控制，需持续检查 | 统计指标若散落在规则中，会影响后续模块扩展 | 将指标汇总作为独立统计步骤，不在规则内部直接写日志或 CSV |
| TD-005 | 后续 C++ 热点迁移暂不设计过深 | 接受延期 | 过早抽象会增加首版复杂度 | v0.1 仅保持模块边界清晰，不引入 C++ 接口层 |
| TD-006 | 运行语义不符合“持续演化”心智模型 | 待处理 | 用户容易把 `--ticks` 理解成目标 tick；恢复旧快照时会误以为从 4000 开始是错误 | `v0.1.1` 增加持续运行、`--steps`、`--until-tick`、暂停和重置 |
| TD-007 | 用户可见文本英文过多 | 待处理 | README、控制台、事件消息不利于中文用户理解模拟结果 | 用户可见文本中文优先，事件保留稳定 ID 但提供中文名 |
| TD-008 | 事件消息缺少具体对象 | 待处理 | “一个谱系”“一个或多个家族”让日志像泛化模板，解释性不足 | 事件 evidence 必须列出谱系 ID、家族 ID、占比、灭绝对象 |
| TD-009 | `latest.json` 并发读写无保护 | 待处理 | 多个进程同时运行/检查可能读到旧快照或互相覆盖 | 增加运行锁或输出目录隔离 |
| TD-010 | 缺少运行总结层 | 待处理 | 用户只能看到事件堆砌，无法判断本次演化走向 | 增加规则化中文总结器，不引入外部 Agent |

## 范围约束

- `v0.1.1` 不实现前生命化学、原始细胞、原核生命、光合作用、氧气环境或宏观生态。
- `v0.1.1` 可以改进事件系统表达，但只能注册 RNA 世界事件。
- 默认不引入 NumPy、Pandas、数据库、GUI 框架或大型仿真框架。

## 下一轮 Leader 检查点

- Builder 是否严格按 `V0.1-design.md` 的 `v0.1.1` 修正要求实现，未扩大生命史阶段范围。
- Reviewer 是否按 `V0.1-review.md` 覆盖快照连续性、随机复现、事件解释性和异常路径。
- 若 Builder 发现设计不可实现，应先产出拆分说明或问题报告，再调整设计文档。
