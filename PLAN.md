# EvoGenesis 概要设计

## 1. 项目定位

`EvoGenesis` 是一个命令行生物演化模拟器。项目从 RNA World 起步，按生命演化顺序逐步扩展，最终成为地球模拟器中的生命模块。

首版 `v0.1` 聚焦 RNA 序列在原始环境中的复制、突变、资源竞争、降解、谱系兴衰、事件日志与快照恢复。整体体验以“科学日志型模拟”为主：用户运行模拟、观察关键事件、检查世界状态，并能从快照继续演化。

## 2. 技术路线

- Python 优先，使用标准库起步。
- 默认不引入重量级依赖；新增依赖必须说明必要性、替代方案和复杂度影响。
- 后续若出现明确性能瓶颈，仅迁移热点模块到 C++，不重写整体项目。
- 开发环境推荐 VS Code，配合 Python、Pylance、Ruff、Even Better TOML；如后续加入 C++，再引入 LLVM、CMake、Ninja 等工具链。

## 3. 总体架构原则

系统按职责分层，避免配置、状态、模拟规则、日志和快照相互硬编码。

- CLI 层：负责命令解析、运行入口、用户输出。
- Engine 层：负责模拟主循环、tick 推进、模块调度、随机源管理。
- Domain State 层：负责世界状态、种群状态、谱系状态、统计指标。
- Rule Module 层：负责具体演化规则，例如 RNA 复制、突变、竞争、降解。
- Event System 层：负责从系统状态中识别里程碑、趋势和灾难事件，并生成可解释日志。
- IO 层：负责配置加载、事件日志、指标输出、快照保存和恢复。

## 4. 事件系统原则

重大事件不应是随机打印的固定文本，而应由系统状态触发。事件的基本模型是：

```text
事件 = 状态阈值 + 时间窗口 + 概率检定 + 影响结果 + 可解释日志
```

事件分三类：

- Milestone：里程碑事件，例如稳定复制子出现、原始细胞出现、蓝藻样谱系出现。
- Trend：趋势事件，例如氧气持续上升、多样性下降、资源长期枯竭。
- Disaster：灾难事件，例如撞击、火山活动、雪球地球、大灭绝。

从 `v0.1` 起就应建立轻量事件框架。首版只需要覆盖 RNA World 的少量事件，但结构必须允许后续版本通过新增事件定义扩展，而不是重写日志系统。

## 5. Version Roadmap

- `v0.1 RNA World`：RNA 复制、突变、资源竞争、日志、快照。
- `v0.2 Prebiotic Chemistry`：小分子池、核苷酸前体、聚合概率、环境催化、RNA 自发生成。
- `v0.3 Protocell`：脂质膜泡、内部 RNA 池、膜稳定性、分裂、死亡。
- `v0.4 Prokaryote`：原核细胞、代谢类型、基因组、种群选择。
- `v0.5 Photosynthesis`：光合作用、氧气积累、环境反馈、大氧化事件。
- `v0.6 Eukaryote`：内共生、线粒体/叶绿体、复杂细胞、有性生殖。
- `v0.7 Multicell`：多细胞、细胞分化、组织复杂度、身体结构。
- `v0.8 Ecology`：物种、生态位、食物链、竞争、共生、疾病。
- `v0.9 MacroEvolution`：适应辐射、大灭绝、长期生命史。
- `v1.0 BioGenesis Engine`：完整 CLI 生物演化引擎，可作为未来地球模拟器的生命模块。

## 6. v0.1 范围边界

`v0.1` 只实现 RNA World。它可以为后续阶段预留概念接口，但不得提前实现前生命化学、原始细胞、原核生命、光合作用或宏观生态。

详细模块、状态字段、事件目录、测试项和验收标准见：

- `docs/builder/designs/V0.1-design.md`
- `docs/reviewer/reviews/V0.1-review.md`

## 7. 总体验收口径

- 可通过命令行启动一次完整模拟。
- 默认配置能产生可读、有变化、可复现的 RNA World 演化过程。
- 日志能解释关键事件发生的原因，而不是只给出结论。
- 快照能保存完整状态并恢复继续运行。
- 代码结构允许后续版本按 roadmap 扩展，而不破坏 `v0.1` 的数据和命令入口。
